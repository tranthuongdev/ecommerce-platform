# ADR-004: Database Schema Conventions

**Date:** 2026-06-25
**Status:** Accepted
**Deciders:** Trần Thưởng
**Supersedes:** None
**Related:** ADR-001 (Modular Monolith), US-002 (Auth schema first usage)

## Context

Sprint 1 introduces the first database schema (auth domain: users, roles, user_roles).
Conventions established here will be applied to all subsequent schemas:

- Sprint 2-3: products, categories, product_variants, cart
- Sprint 4-5: orders, order_items, payments, inventory, inventory_reservations
- Sprint 8-9: outbox, processed_events (Kafka outbox pattern)
- Sprint 17: refresh_tokens (when properly persisted)

Total estimated tables by end of Year 1: ~25-30.

Inconsistent conventions across these tables will cause:
- Cognitive overhead for every query
- Bugs from convention mismatch (audit field naming differs per table)
- Painful refactoring when migrating to microservices in Year 2
- Difficulty when extracting shared logic to common module

This ADR locks down conventions early to avoid these issues.

## Decision

The following conventions are mandatory for all new schema work in Year 1.
Deviations require a new ADR documenting the exception.

### 1. Primary Key Strategy

**Use UUID v4 generated by Postgres `gen_random_uuid()`.**

- Type: `UUID NOT NULL PRIMARY KEY`
- Generated by DB on insert if not provided
- Java entity may pre-generate via `@PrePersist` for offline-first scenarios

Future migration path: UUID v7 (time-ordered) when Postgres 17 native function
becomes stable in our usage. Evaluation deferred to Sprint 13 (Q3) alongside
EXPLAIN ANALYZE deep-dive work.

### 2. Naming Convention

- Tables: lowercase snake_case, plural (`users`, `order_items`)
- Columns: lowercase snake_case (`created_at`, `email_verified_at`)
- Indexes: `idx_<table>_<columns>` or `uq_<table>_<columns>` for unique
- Foreign keys: `fk_<table>_<referenced_table>`
- Check constraints: `chk_<table>_<rule>`
- Java entities: PascalCase (User), fields: camelCase (createdAt)

Mapping handled by Hibernate's `PhysicalNamingStrategyStandardImpl` (default for snake_case ↔ camelCase).

### 3. Audit Fields

Every business entity must extend `BaseEntity` (defined in common module), providing:

- `id` UUID, primary key
- `created_at` TIMESTAMP, set on insert, immutable
- `created_by` UUID, set on insert from SecurityContext
- `updated_at` TIMESTAMP, updated on every save
- `updated_by` UUID, updated on every save
- `version` BIGINT, optimistic locking via `@Version`

**Exceptions:** Lookup/reference tables (e.g., `roles`) may use minimal audit
(only `created_at`) since they are mostly static data managed via migrations.

### 4. Soft Delete Strategy

Soft delete only when there's a clear business or compliance reason:

- ✅ `users` — GDPR, audit trail of who registered when
- ✅ `orders` — accounting, dispute resolution (future, Sprint 4)
- ❌ `cart_items` — ephemeral data, hard delete
- ❌ `refresh_tokens` — security, hard delete on revoke
- ❌ `outbox_events` — operational, hard delete after relay

Implementation:
- Add nullable `deleted_at TIMESTAMP` column
- Use Hibernate 6.3+ `@SQLRestriction("deleted_at IS NULL")` on entity
- Repository methods automatically filter

### 5. Email & String Uniqueness

When uniqueness must be case-insensitive (e.g., email):

1. **Application layer:** normalize (trim + lowercase) before save
2. **Database layer:** functional unique index on `LOWER(column)`

Defense in depth — both layers enforce, neither alone is sufficient.

### 6. Foreign Key Cascade Rules

- `ON DELETE CASCADE` — when child cannot exist without parent (e.g., `user_roles` → `users`)
- `ON DELETE RESTRICT` — when accidental deletion must be prevented (e.g., `user_roles` → `roles`)
- `ON DELETE SET NULL` — rarely used, only when relationship is truly optional

**Default if unsure:** RESTRICT. Forces explicit decision in application code.

### 7. Index from Day 1

Every migration must include indexes for:

- Primary keys (auto)
- Unique constraints (auto)
- **Foreign keys** (NOT automatic in Postgres — must add explicitly)
- Columns used in WHERE clauses for hot queries
- Columns used in ORDER BY for paginated lists

Use partial indexes for filtered queries:
- `CREATE INDEX idx_users_active ON users (id) WHERE deleted_at IS NULL`

Adding indexes retroactively requires:
- New migration file
- Production: `CREATE INDEX CONCURRENTLY` (non-blocking but slow)
- Avoid by indexing upfront based on access pattern analysis.

### 8. Migration File Conventions

- Location: `<module>/src/main/resources/db/migration/`
- Naming: `V<n>__<description>.sql` (double underscore)
- Version numbers: sequential integers (V1, V2, V3...), never decimals
- One logical change per file
- **Never modify a merged migration** — create a new migration to fix

Flyway `baseline-on-migrate: true` enabled for safety, but never relied on.

## Consequences

### Positive

- Consistent patterns across all entities — onboarding (or 6-month-later self) understands any table immediately
- Schema ready for Year 2 scaling: UUID for multi-tenant, soft delete for audit
- DB-level enforcement of business rules (defense in depth)
- Index strategy from day 1 avoids retroactive migration pain
- Foundation for `BaseEntity` reuse across modules — less boilerplate per entity

### Negative

- Boilerplate per entity (mitigated by `BaseEntity`)
- UUID v4 fragmentation at scale (mitigated by Q3 v7 evaluation)
- Soft delete adds query complexity (mitigated by `@SQLRestriction`)
- `BaseEntity` couples all modules to `common` (acceptable, already the design)

### Neutral

- Conventions are opinionated — some teams prefer BIGINT, hard-delete-only, etc.
  Documented here so future deviations can be debated against stated rationale.

## Alternatives Considered

### Alt 1: BIGINT auto-increment primary key
- **Rejected:** Year 2 multi-tenant requires distributed ID generation; UUID is the path

### Alt 2: Single-role enum on users table
- **Rejected:** E-commerce users routinely need multi-role (buyer + seller); RBAC pattern needed

### Alt 3: Hard delete only (audit via separate event log)
- **Rejected:** Operational complexity higher than soft delete for current scale

### Alt 4: No audit fields, rely on application logging
- **Rejected:** Logs are queryable as time-series, not as entity history; debugging requires both

### Alt 5: CITEXT extension for case-insensitive email
- **Rejected:** Functional index achieves same result without extension dependency
